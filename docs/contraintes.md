\# Contraintes du projet



\## Contexte général



Le projet est réalisé dans un cadre universitaire.



Il doit donc être :

\- faisable par un étudiant

\- cohérent techniquement

\- maintenable dans le temps

\- suffisamment riche pour démontrer des compétences solides

\- réaliste en termes de charge de travail



\---



\## Contraintes techniques



\### Front-end

Le front-end doit être développé uniquement avec :

\- HTML

\- CSS

\- JavaScript vanilla



Les frameworks front-end comme React ou Vue.js ne doivent pas être utilisés.



\### Architecture

Le projet doit reposer sur une architecture claire et structurée.



Une architecture de type MVC ou MVP doit être privilégiée selon les besoins, avec une justification simple et cohérente du choix.



Le code doit être organisé de manière à :

\- séparer les responsabilités

\- isoler facilement les erreurs

\- éviter le code spaghetti

\- faciliter la maintenance



\### Back-end

Le back-end doit :

\- contenir la logique métier

\- gérer les permissions

\- valider les données

\- exposer une API REST



\### Base de données

Le projet doit utiliser une base de données relationnelle cohérente avec les besoins fonctionnels.



Les relations entre entités doivent être clairement définies.



\---



\## Contraintes fonctionnelles



Le projet doit respecter la distinction entre :

\- administration globale du hackathon

\- administration locale de l’équipe



Cela signifie notamment que :

\- l’organisateur du hackathon gère l’événement global

\- le chef d’équipe gère l’organisation interne de son équipe

\- les permissions ne doivent pas être mélangées



Le cœur du projet repose sur :

\- la gestion des hackathons

\- la gestion des équipes

\- la gestion des tâches

\- la gestion des rôles et permissions



\---



\## Contraintes de conception



Le projet doit rester :

\- lisible

\- sobre

\- moderne

\- intuitif

\- maintenable



Il faut éviter :

\- la surcharge fonctionnelle en première version

\- les fonctionnalités gadgets trop tôt

\- une architecture inutilement complexe

\- les technologies non demandées



\---



\## Contraintes de progression



Le projet doit être développé par versions progressives :

\- V1 : socle minimal fonctionnel

\- V2 : enrichissement logique

\- V3 : amélioration avancée

\- V4 : bonus



La V1 doit rester volontairement simple et réaliste.



\---



\## Contraintes de qualité



Le projet doit viser :

\- une bonne cohérence globale

\- un code propre

\- une structure claire

\- des choix techniques justifiés

\- une application crédible académiquement
